# Tessera v0.1 Plan

日期：2026-05-14

## 1. 目标

v0.1 的目标是建立一个最小但真实的本地 runtime，而不是先做一个漂亮但不可持续的聊天 TUI。

v0.1 必须证明四件事：

- 同一套 headless core 可以同时被 CLI 和 TUI 使用。
- Provider 差异不会污染核心协议。
- 每次运行都能生成可审计、可回放的 trace。
- 后续工具、agent、memory、skill、swarm 能接入现有边界，而不是要求重写 v0.1。
- Provider capability、reasoning、cache、usage、cost 等体验数据能进入标准协议和 trace，而不是散落在 UI 或 provider adapter。

## 2. 产品范围

v0.1 面向单用户、本地运行、手动交互。

必须支持：

- OpenAI-compatible provider 的流式聊天。
- Ollama provider 的流式聊天。
- Mock provider，用于测试、fixture 和 replay。
- Provider capability schema。
- Reasoning delta 的可选协议支持。
- Token usage、cache telemetry、latency 和 cost estimate 的可选 trace 字段。
- `tessera chat` headless 对话。
- `tessera doctor --json` 机器可读健康检查。
- 最小 Ratatui TUI：单个主聊天窗口、模型/profile 切换入口、基础保存/导出入口。
- 本地 session 保存。
- JSONL trace。
- SQLite 索引，用于列出 thread、turn、item、task 和 artifact。
- 基础 `Thread` / `Turn` / `Item` / `Task` / `Artifact` / `EventFrame` schema。
- provider mock、storage、core run loop 的基础测试。

明确不做：

- 自动执行 shell。
- 文件修改工具。
- MCP runtime。
- 多 agent runtime。
- swarm scheduler。
- 长期 memory runtime。
- learning runtime。
- Auto model router。
- YOLO / trusted workspace 自动批准模式。
- 复杂多窗口布局。
- HTTP/SSE runtime API 的完整实现。
- 插件市场、云同步、账号体系。

## 3. v0.1 Crate 边界

v0.1 只开必要 crate，避免一开始被长期愿景拖重。

建议首批 crate：

```text
crates/
  protocol/
  client/
  core/
  providers/
  storage/
  config/
  cli/
  tui/
```

`client` 是用户明确优先 GUI 准备后进入 v0.1 的窄边界 crate：只承载 UI-neutral intent 和 projection，不拥有 runtime。

可作为模块先存在、暂不独立开 crate：

```text
core::tasks
core::windows
core::policy_types
core::tool_types
core::skill_types
core::agent_types
core::memory_types
core::learning_types
core::swarm_types
```

暂不开 crate：

```text
tools/
policy/
tasks/
windows/
agents/
memory/
skills/
swarm/
learning/
runtime_api/
diagnostics/
snapshots/
```

这些能力必须先有稳定协议和测试夹具，再独立成 crate。

## 4. 交付阶段

### Phase 0：架构冻结

交付：

- `docs/technical-architecture.md`
- `docs/deepseek-tui-lessons.md`
- `docs/v0.1-plan.md`
- `docs/protocol-v0.md`
- `docs/trace-schema-v0.md`
- `docs/crate-boundaries.md`
- `AGENTS.md`

完成条件：

- v0.1 必做和不做边界清楚。
- 核心类型和 trace 结构足以开始 Rust scaffold。
- AI 开发边界写进仓库，后续不会只靠口头约束。

### Phase 1：Workspace Scaffold

交付：

- Rust workspace。
- 首批 crate。
- crate README。
- 基础 CI。
- `cargo fmt --check`、`cargo clippy`、`cargo test` 可运行。

完成条件：

- 没有业务功能也能通过基础质量门禁。
- crate 之间依赖方向符合 `docs/crate-boundaries.md`。

### Phase 2：Protocol And Storage

交付：

- `protocol` 中的 ID、Thread、Turn、Item、Task、Artifact、EventFrame。
- Provider capability、reasoning delta、extended usage、route decision 占位。
- JSONL trace writer。
- SQLite schema 和 repository。
- schema version。
- storage 单元测试。

完成条件：

- 可以写入一段模拟 chat trace。
- 可以从 SQLite 列出 thread/turn/item。
- JSONL 和 SQLite 记录能通过同一个 trace id 关联。
- Trace 能记录 usage、latency、cache telemetry、cost estimate 的可选字段。

### Phase 3：Provider And Core

交付：

- Provider trait。
- OpenAI-compatible adapter。
- Ollama adapter。
- Mock provider。
- ConversationEngine。
- provider error normalization。
- provider capability discovery。
- provider extension metadata redaction。

完成条件：

- mock provider 可以驱动完整 run loop。
- provider adapter 只输出标准事件，不接触 storage 和 TUI。
- core 可以把 provider stream 转成 EventFrame 并写入 trace。
- provider 私有 metadata 只能以安全 extension 进入 trace。

### Phase 4：CLI

交付：

- `tessera doctor --json`
- `tessera chat`
- 基础 config 读取。
- secret 只从 env 读取。
- doctor 输出 provider profile、data dir、trace writable、SQLite index health。

完成条件：

- CLI 可以用 mock provider 完成离线对话。
- CLI 可以用 OpenAI-compatible 或 Ollama 完成真实流式对话。
- 每次运行都有 trace。

### Phase 5：TUI

交付：

- 最小 Ratatui chat。
- 模型/profile 切换入口。
- reasoning delta 展示占位。
- usage/cache/cost 状态展示占位。
- `/new`、`/save`、`/export`。
- 单窗口模型和基础 WindowId。

完成条件：

- TUI 不直接依赖 provider SDK。
- TUI 只订阅 core 事件和发送用户输入。
- 关闭 TUI 不破坏已保存 session/trace。

### Phase 6：Replay Gate

交付：

- mock trace fixture。
- replay runner 初版。
- golden trace test。

完成条件：

- 真实 API 失败时，可以用 fixture 重放核心行为。
- AI 后续修改 core/provider/storage 时，有稳定回归样本。

## 5. 质量门禁

v0.1 实现完成前，必须能稳定运行：

```bash
cargo fmt --check
cargo clippy --workspace --all-targets -- -D warnings
cargo test --workspace
tessera doctor --json
tessera chat --provider mock --prompt "hello"
```

如果引入真实 provider smoke test，必须默认跳过，只有在对应环境变量存在时才运行。

## 6. 稳定演进原则

- 先固定协议，再扩展 UI。
- 先有 mock/replay，再接复杂 provider。
- 先有 trace，再做 tool/agent。
- 先把 provider capability、usage、cache、cost 记录好，再做 Auto router。
- 先有单 agent 语义，再做多 agent。
- 先有 handoff 和 reviewer gate，再做 swarm。
- 先有 sandbox/checkpoint，再允许文件修改工具。
- memory 第一版可以简单，但 schema 不能锁死成全局 Markdown。
- skill 第一版先兼容 `SKILL.md`，高级 `skill.toml` 后置。
- 所有未来能力只能通过 `protocol` 和 `core` 的公开边界进入系统。

## 7. 不做的诱惑

v0.1 最容易失控的地方是提前做“看起来很有产品感”的功能。下面这些即使很诱人，也应后置：

- 多 pane / tiling window manager。
- 自动代码修改。
- shell tool。
- Auto model router。
- YOLO 模式。
- MCP 市场。
- skill 执行 runtime。
- 长期记忆搜索。
- 多 agent 并发。
- learning 自动改 prompt。
- cost dashboard。
- desktop/editor integration。

这些不是否定方向，而是防止第一版 runtime 还没站稳就被未来能力拉散。
